# SOL Meta Model v1.0
**Signature: AAM-V2_ARTSYBASHEV_UA_KHARKIV_METAMODEL**

## 1. Введение
Настоящий документ определяет **Метамодель** (Meta Model) языка SOL. В отличие от онтологии, которая определяет "что существует", метамодель определяет "как это устроено" внутри системы. Она служит фундаментом для создания абстрактного синтаксического дерева (AST) и внутреннего представления (IR).

## 2. Базовая структура (Generic Node)
Все сущности в SOL наследуют базовую структуру метаданных, обеспечивающую прослеживаемость и уникальность.

| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `uid` | UUID/String | Глобально уникальный идентификатор экземпляра |
| `id` | String | Локальный человекочитаемый идентификатор (код) |
| `type` | Enum | Тип сущности из SOL Core Ontology |
| `name` | String | Краткое наименование на естественном языке |
| `specification` | Link | Ссылка на `TYPE_DEFINITION` (шаблон) |
| `parent` | Link | Ссылка на родительский контейнер (иерархия) |
| `doc_link` | URI | Ссылка на внешнюю документацию/стандарт |
| `metadata` | Map | Произвольные пары ключ-значение для расширения |

## 3. Спецификация структур по сущностям

### 3.1 TYPE_DEFINITION (Specification) [NEW]
Шаблон или класс для создания экземпляров.
- **Fields:**
  - `target_type`: Тип сущности, для которой создан шаблон.
  - `schema`: Определение ожидаемых свойств и связей.
  - `constraints`: Глобальные ограничения для всех наследников.

### 3.2 ENTITY (Сущность)
Физический или логический объект.
- **Fields:**
  - `properties`: Список ссылок на `PROPERTY`.
  - `relations`: Список ссылок на `RELATION`.
  - `state`: Текущая ссылка на `STATE`.

### 3.3 PROPERTY (Свойство)
Качественная или количественная характеристика.
- **Fields:**
  - `value_type`: Тип данных (bool, float, string, object).
  - `unit`: Единица измерения (опционально).
  - `default`: Значение по умолчанию.

### 3.4 RELATION (Связь)
Топологическая или логическая зависимость.
- **Fields:**
  - `source`: UID источника.
  - `target`: UID цели.
  - `relation_kind`: Характер связи (ParentOf, ConnectedTo, MountsOn).

### 3.5 CONSTRAINT (Ограничение)
Граница допустимых значений или состояний.
- **Fields:**
  - `expression`: Формализованное логическое выражение.
  - `severity`: Уровень критичности (Hard, Soft, Advisory).
  - `applies_to`: Список UID/Types.

### 3.6 STATE (Состояние)
Статическое положение системы в моменте.
- **Fields:**
  - `active_constraints`: Ограничения, действующие в этом состоянии.
  - `duration`: Ожидаемое время пребывания (опционально).

### 3.7 EVENT (Событие)
Мгновенный переход или сигнал.
- **Fields:**
  - `trigger`: Условие возникновения.
  - `payload`: Данные, передаваемые событием.
  - `origin`: Источник возникновения.

### 3.8 PROCESS (Процесс)
Последовательность действий (Will/Execution).
- **Fields:**
  - `steps`: Список вложенных операций.
  - `resources_required`: Список ссылок на `RESOURCE`.
  - `capabilities_used`: Список ссылок на `CAPABILITY`.

### 3.9 RESOURCE (Ресурс)
Материальное/энергетическое обеспечение.
- **Fields:**
  - `availability`: Статус доступности.
  - `capacity`: Максимальный объем/пропускная способность.

### 3.10 CAPABILITY (Возможность)
Способность выполнять определенный класс задач.
- **Fields:**
  - `level`: Уровень компетенции/производительности.
  - `interfaces`: Описание входных и выходных портов.

### 3.11 MEASUREMENT (Измерение)
Актуальное значение, полученное из реальности.
- **Fields:**
  - `timestamp`: Время замера.
  - `actual_value`: Значение.
  - `precision`: Погрешность/Точность.
  - `source_device`: Ссылка на прибор/сенсор.

## 4. Иерархия: Instance vs Specification
Метамодель поддерживает строгое разделение:
1. **Specification Level:** Типовое описание ("Подшипник качения серии 6200").
2. **Instance Level:** Конкретный объект с уникальным серийным номером ("Подшипник №12345 на валу А").

## 5. Итог
Структура Метамодели инвариантна к формату хранения. Она определяет логическую схему объектов в оперативной памяти Runtime и структуру узлов в AST.

## 6. COMPILATION_GUNA_AXIOMS
Назначение "веса" и гуны для сущностей в процессе компиляции и исполнения.

| Категория | Гуна | Определение | Весовой коэффициент (W) |
| :--- | :--- | :--- | :--- |
| **OBJ (Entity)** | **Sattva** | Мета-структура, описание, ясность. | 0.1 - 0.3 (Высокая мобильность) |
| **OP (Primitive)** | **Rajas** | Активность, трансформация, энергия. | 0.4 - 0.7 (Динамическое равновесие) |
| **CON (Connection)** | **Tamas** | Инерция, физическая связь, плотность. | 0.8 - 1.0 (Максимальная стабильность) |

### Аксиомы веса:
1. **Inertia Rule:** Узлы `ConnectionNode` (Tamas) имеют наивысший вес, так как они фиксируют топологию и сопротивляются изменениям.
2. **Activity Rule:** Узлы `DefinitionNode` (Rajas) обладают переменным весом в зависимости от сложности вложенных `dhatu`.
3. **Clarity Rule:** Узлы `RootNode` и мета-спецификации (Sattva) имеют минимальный вес, обеспечивая "скелет" системы без внесения инерции.
